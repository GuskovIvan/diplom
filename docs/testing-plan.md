# План И Результаты Тестирования

## Автоматическая Проверка

Запуск:

```bash
npm test
```

Текущее состояние автоматических тестов:

```text
Test Files: 2 passed
Tests:      8 passed
```

Покрытые сценарии:

- расчет позиции карточки при перемещении в пустую колонку;
- добавление карточки в конец колонки;
- вставка карточки перед другой карточкой;
- игнорирование перемещаемой карточки при расчете новой позиции;
- создание событий `MEMBER_ADDED`, `MEMBER_UPDATED`, `MEMBER_REMOVED`;
- создание событий `GROUP_CREATED`, `GROUP_UPDATED`, `GROUP_DELETED`;
- увеличение `version` задачи при перемещении;
- фиксация устаревшего `clientVersion` как разрешенного конфликта;
- перемещение карточки внутри одной колонки;
- доставка `sync:event` через Socket.IO авторизованному участнику проекта.

## Сборка

Запуск:

```bash
npm run build
```

Проверяется:

- компиляция backend TypeScript;
- сборка frontend через Vite;
- отсутствие TypeScript-ошибок в production-коде;
- отсутствие тестовых файлов в production-сборке сервера.

## Ручные Функциональные Проверки

1. Регистрация нового пользователя.
2. Вход демо-пользователя.
3. Создание проекта.
4. Добавление участника.
5. Изменение роли участника.
6. Создание группы исполнителей.
7. Создание колонки.
8. Создание карточки.
9. Назначение карточки пользователю или группе.
10. Редактирование карточки.
11. Перемещение карточки между колонками.
12. Перемещение карточки внутри одной колонки.
13. Удаление карточки.
14. Просмотр журнала событий.

## Проверка Ролей

`OWNER`:

- добавляет администратора;
- изменяет роль участника;
- создает группу из администратора и участника;
- управляет задачами и колонками.

`ADMIN`:

- добавляет обычного участника;
- создает группу только из обычных участников;
- управляет задачами и колонками;
- не назначает задачи владельцу.

`MEMBER`:

- видит доску;
- перемещает назначенную ему задачу;
- не создает колонки;
- не удаляет чужие задачи;
- не управляет участниками.

## Проверка Синхронизации

1. Открыть две вкладки браузера.
2. Войти владельцем в первой вкладке.
3. Войти исполнителем во второй вкладке.
4. В первой вкладке назначить исполнителю задачу.
5. Убедиться, что во второй вкладке обновилась доска.
6. Переместить карточку.
7. Проверить появление события `TASK_MOVED`.
8. Добавить участника или группу.
9. Проверить появление событий `MEMBER_ADDED` или `GROUP_CREATED`.

## Проверка Конфликтов

1. Открыть две вкладки.
2. В обеих вкладках загрузить одну карточку.
3. В первой вкладке изменить карточку или переместить ее.
4. Во второй вкладке выполнить действие со старой версией карточки.
5. Проверить ответ `conflictResolved: true`.
6. Проверить событие с `strategy: LAST_WRITE_WINS`.

## Измерение Задержки

Клиент измеряет задержку двумя способами:

- по разнице между временем получения `sync:event` и `serverTime`;
- по Socket.IO `ping`/`ack` каждые 5 секунд.

Для текста ВКР нужно зафиксировать не менее 20-30 измерений.

Рекомендуемая таблица:

| Количество клиентов | Средняя задержка, мс | Min, мс | Max, мс | Потеря событий |
| --- | ---: | ---: | ---: | ---: |
| 1 | заполнить | заполнить | заполнить | 0 |
| 5 | заполнить | заполнить | заполнить | 0 |
| 10 | заполнить | заполнить | заполнить | 0 |

## Нагрузочная Проверка

Минимальный сценарий:

1. Открыть 5-10 вкладок или браузерных профилей.
2. Выполнить серию перемещений карточек.
3. Зафиксировать отсутствие расхождения состояния доски.
4. Проверить, что журнал событий содержит все операции.

Расширенный сценарий:

1. Написать k6 или Artillery-сценарий для REST-операций.
2. Отдельно проверить Socket.IO через несколько параллельных клиентов.
3. Сравнить среднюю задержку при 1, 10, 25 и 50 клиентах.

## Вывод Для Защиты

Практическая часть проверяется не только вручную. В проект добавлены автоматические интеграционные тесты, которые используют реальную PostgreSQL-базу и реальное Socket.IO-подключение. Это подтверждает, что синхронизация, журнал событий и разрешение конфликтов работают на уровне backend-логики, а не только в интерфейсе.
