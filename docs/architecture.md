# Архитектура Практической Части

## Назначение

Приложение предназначено для командного управления проектными задачами по Kanban-модели. Главная исследовательская идея практической части: разделить надежное изменение данных и быструю доставку уведомлений.

- REST API отвечает за проверку прав, транзакции и сохранение состояния.
- PostgreSQL является источником истины.
- WebSocket отвечает за оперативную синхронизацию открытых вкладок.

## Границы Функциональности

Чтобы практическая часть выглядела цельно, функциональность разделена на основной контур и поддерживающие представления.

Основной контур:

- Kanban-доска;
- роли `ADMIN`, `MEMBER`;
- назначение задач участникам и группам;
- серверный журнал `TaskEvent`;
- WebSocket-синхронизация;
- версионирование задач и фиксация конфликтов.

Поддерживающие представления:

- `Roadmap` использует те же колонки и задачи, но показывает их как маршрут проекта;
- `Объявления` показывают адресную коммуникацию внутри проекта;
- `Аудит` выводит события, метрики и признаки корректной realtime-синхронизации.

Так дополнительные экраны не расширяют предметную область хаотично, а помогают продемонстрировать надежность, доступы и синхронизацию основной Kanban-модели.

## Компоненты

- React-клиент отображает проекты, команду, группы, колонки, карточки, журнал событий и WebSocket-статус.
- Express REST API принимает команды: регистрация, вход, управление проектами, участниками, группами, колонками и задачами.
- Prisma ORM описывает модель данных и выполняет миграции.
- PostgreSQL хранит пользователей, проекты, роли, задачи, версии и журнал событий.
- Socket.IO создает защищенные комнаты проекта и доставляет события участникам.

## Модель Данных

Ключевые сущности:

- `User` - пользователь системы.
- `Project` - рабочее пространство команды.
- `ProjectMember` - связь пользователя с проектом и ролью.
- `TaskGroup` и `TaskGroupMember` - группа исполнителей внутри проекта.
- `BoardColumn` - колонка Kanban-доски.
- `Task` - карточка задачи с приоритетом, позицией и версией.
- `TaskEvent` - серверный журнал событий для синхронизации и аудита.

Событийная модель охватывает проектные действия:

```text
PROJECT_CREATED
MEMBER_ADDED, MEMBER_UPDATED, MEMBER_REMOVED
GROUP_CREATED, GROUP_UPDATED, GROUP_DELETED
COLUMN_CREATED, COLUMN_UPDATED, COLUMN_DELETED
TASK_CREATED, TASK_UPDATED, TASK_MOVED, TASK_DELETED
```

## Роли

`ADMIN`:

- управляет администраторами и участниками;
- создает группы с администраторами и участниками;
- управляет колонками и задачами.

`MEMBER`:

- видит доску проекта;
- перемещает только задачи, назначенные ему лично или его группе;
- не управляет участниками, группами, колонками и чужими задачами.

## Поток Изменения Карточки

1. Пользователь перетаскивает карточку.
2. Клиент делает оптимистичное изменение UI.
3. Клиент отправляет `POST /api/tasks/:taskId/move` с `columnId`, `beforeTaskId` или `afterTaskId`, а также `clientVersion`.
4. Сервер проверяет JWT и членство пользователя в проекте.
5. Сервер проверяет право перемещения задачи.
6. Сервер проверяет, что целевая колонка и якорная карточка принадлежат проекту.
7. Сервер вычисляет новую дробную позицию без полного пересчета всей колонки.
8. Сервер увеличивает `version` задачи и сохраняет изменение в PostgreSQL.
9. Сервер создает `TaskEvent` типа `TASK_MOVED`.
10. Socket.IO публикует `sync:event` в комнату `project:{projectId}`.
11. Другие вкладки получают событие и загружают актуальное состояние через REST API.

## Разрешение Конфликтов

Для карточек используется стратегия `LAST_WRITE_WINS` с версионированием. Каждая карточка имеет поле `version`. Клиент передает версию, на основе которой было выполнено действие.

Если сервер получает устаревший `clientVersion`, операция применяется как последнее действие, но событие фиксирует конфликт:

```json
{
  "conflict": {
    "resolved": true,
    "strategy": "LAST_WRITE_WINS",
    "clientVersion": 1,
    "previousServerVersion": 2
  }
}
```

Для Kanban-доски это оправдано: пользовательские операции происходят над отдельными карточками, а не над посимвольным текстом. Поэтому нет необходимости внедрять CRDT или Operational Transformation, которые сложнее объяснять и поддерживать в рамках бакалаврской работы.

## REST И WebSocket

REST используется для команд, которые требуют авторизации, валидации и транзакционного сохранения:

- регистрация и вход;
- создание проекта;
- управление участниками и группами;
- создание, изменение и удаление колонок;
- создание, изменение, удаление и перемещение карточек;
- получение полного состояния доски и журнала событий.

WebSocket используется для доставки уведомлений:

- изменения участников и групп;
- изменения колонок и карточек;
- presence подключенных пользователей;
- измерение задержки через ping/ack.

## Масштабирование

В текущей реализации Socket.IO-комнаты хранятся в памяти одного Node.js-процесса. Этого достаточно для учебного стенда и защиты.

Для промышленного масштабирования нужно подключить Socket.IO Redis adapter:

1. несколько backend-инстансов подключаются к Redis;
2. REST-инстанс после транзакции публикует событие;
3. все WebSocket-инстансы получают событие через Redis Pub/Sub;
4. каждый инстанс доставляет событие своим подключенным клиентам.

Redis уже описан в `docker-compose.yml`, но в коде adapter намеренно не подключен. Это честно относится к разделу перспектив развития.

## Что Доказывает Компетентность

- Использование транзакций для изменений состояния.
- Отдельная таблица событий для аудита и синхронизации.
- Проверка ролей на сервере, а не только в интерфейсе.
- Версионирование карточек и фиксация конфликтов.
- Интеграционные тесты с реальной PostgreSQL-базой.
- Интеграционный тест доставки Socket.IO-события.
