# Realtime Kanban для дипломной работы

Практическая часть выпускной квалификационной работы:

```text
Разработка веб-приложения для командного управления проектами
с синхронизацией доски задач в реальном времени
на основе REST API и WebSocket
```

Приложение реализует Kanban-доску для командной работы: REST API принимает проверяемые команды изменения данных, PostgreSQL хранит состояние и журнал событий, Socket.IO доставляет изменения всем подключенным участникам проекта без ручного обновления страницы.

## Продуктовая Рамка Для Диплома

Главный сценарий проекта - совместная работа нескольких участников с одной Kanban-доской в реальном времени.

Функции сгруппированы вокруг этой идеи:

- `Доска` - основной рабочий экран: колонки, карточки, назначение исполнителей и drag-and-drop.
- `Roadmap` - дополнительное представление тех же задач по этапам проекта, без отдельной бизнес-модели.
- `Объявления` - коммуникационный модуль проекта с адресной доставкой участникам.
- `Аудит` - журнал серверных событий, метрики синхронизации, задержка WebSocket и фиксация конфликтов версий.

Так проект демонстрирует не набор разрозненных функций, а цельную систему: REST отвечает за надежное изменение состояния, PostgreSQL - за хранение и аудит, WebSocket - за оперативную доставку изменений, роли - за контроль доступа.

## Стек

- Backend: Node.js, TypeScript, Express, Socket.IO.
- Frontend: React, TypeScript, Vite, dnd-kit.
- Database: PostgreSQL, Prisma ORM, Prisma migrations.
- Auth: JWT + bcrypt.
- Dev environment: Docker Compose для PostgreSQL; Redis оставлен как инфраструктурная заготовка для масштабирования Socket.IO через Redis adapter.

## Что Реализовано

- Регистрация и вход пользователей.
- Проекты с участниками и ролями `ADMIN`, `MEMBER`.
- Управление участниками проекта и группами исполнителей.
- Создание, изменение, удаление и перемещение колонок и задач.
- Назначение задачи одному исполнителю или группе.
- Ограничения доступа: исполнитель может перемещать только назначенные ему или его группе задачи.
- Журнал событий `TaskEvent` для проекта.
- WebSocket-комнаты `project:{projectId}` и presence подключенных пользователей.
- Версионирование карточек и фиксация конфликтов по стратегии `LAST_WRITE_WINS`.
- Автоматические unit и integration tests для позиционирования, событий, конфликтов и доставки Socket.IO.

## Быстрый Запуск

1. Установить зависимости:

```bash
npm install
```

2. Создать `.env` из примера:

```bash
copy .env.example .env
```

3. Поднять PostgreSQL.

Через Docker:

```bash
docker compose up -d postgres
```

Или локально создать БД `kanban_diplom`, пользователя `kanban` с паролем `kanban_password` и проверить `DATABASE_URL` в `.env`.

4. Применить миграции и создать демо-данные:

```bash
npm run prisma:deploy
npm run db:seed
```

5. Запустить приложение:

```bash
npm run dev
```

Клиент: `http://localhost:5173`  
API: `http://localhost:4000/api`

Для просмотра production-сборки:

```bash
npm run build
npm run start
```

Открыть: `http://localhost:4000`

## Проверка Качества

```bash
npm test
npm run build
```

Тесты проверяют:

- алгоритм расчета позиции карточки при перемещении;
- создание событий для участников и групп;
- увеличение версии задачи при перемещении;
- обнаружение устаревшего `clientVersion`;
- перемещение внутри одной колонки;
- доставку `sync:event` через Socket.IO подключенному участнику проекта.

## Архитектурная Идея

REST API является авторитетным источником изменений. Клиент отправляет команду, сервер проверяет JWT, роль пользователя, принадлежность сущностей проекту, выполняет транзакцию, увеличивает версию измененной сущности и сохраняет событие в `TaskEvent`.

WebSocket не заменяет REST и базу данных. Он доставляет легкие уведомления о том, что состояние проекта изменилось. Получив событие, клиент загружает актуальную доску из REST API.

События проекта:

- `PROJECT_CREATED`
- `MEMBER_ADDED`, `MEMBER_UPDATED`, `MEMBER_REMOVED`
- `GROUP_CREATED`, `GROUP_UPDATED`, `GROUP_DELETED`
- `COLUMN_CREATED`, `COLUMN_UPDATED`, `COLUMN_DELETED`
- `TASK_CREATED`, `TASK_UPDATED`, `TASK_MOVED`, `TASK_DELETED`

## Что Показывать На Защите

1. Открыть приложение в двух вкладках.
2. Войти администратором в первой вкладке и исполнителем во второй.
3. Показать роли и ограничения: исполнитель не управляет чужими задачами.
4. Создать или изменить задачу и показать событие в журнале.
5. Перетащить карточку и показать обновление второй вкладки без перезагрузки.
6. Добавить участника или группу и показать, что изменение тоже попадает в журнал событий.
7. Открыть `http://localhost:4000/api/health` и показать PostgreSQL.
8. Запустить `npm test` и показать автоматические проверки.
9. Открыть `prisma/schema.prisma` и объяснить модель данных.

## Ограничения

- Redis adapter не подключен в коде, потому что локальная версия рассчитана на один backend-процесс. Redis в `docker-compose.yml` оставлен для раздела перспектив масштабирования.
- JWT хранится в `localStorage`, что удобно для учебной демонстрации, но для промышленного варианта лучше рассмотреть httpOnly cookie и CSRF-защиту.
- Нагрузочные измерения нужно фиксировать отдельно в тексте ВКР: средняя задержка, минимум, максимум, число клиентов, параметры стенда.
